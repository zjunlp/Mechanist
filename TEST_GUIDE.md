# MECHANICA 测试注意事项

本文档列出使用 MECHANICA 插件时需要注意观察的关键行为。在你自己的实验任务运行 `/auto` 的过程中，留意以下方面：

---

## 一、task.md 编写与内容处理

`task.md` 是每个项目工作目录下的**任务说明书**，使用自然语言编写，需要人工编写并提供。

### 1.1 task.md 应该包含什么

正文自由发挥，必填项随 `behavior-source` / `mechanism` 参数组合而变：

| 内容 | 何时必需 | 说明 |
|------|----------|------|
| **behavior** | `behavior-source: given` / `given-validation` | 提供一个具体、可证伪的现象 |
| **topic** | `behavior-source: discovery` | 提供一个想研究的大方向，细节现由 Mechanist 自己挖掘。 |
| **family** | `mechanism: given` | 提供一个具体机理方法（如 Fisher information / steering vectors）。 |
| **model / data** | 推荐 | 实验用的模型与数据（写清路径）。。 |
| **claim 列表 / goal** | 可选 | 你希望验证的若干条断言以及本轮目标。 |

### 1.2 声明计算资源（GPU 预算，非必须）

用自然语言写明**实验阶段的 GPU 小时数预算和卡数预算**即可，系统会将其注入实验流程进行监控。例如：

> 你有 8 小时的 GPU 预算，在 GPU 用时达到预算前，不要以 GPU 预算为理由暂停或简化实验。你同时最多只能占用 8 张 GPU 中的 4 张。

要点：
- **充足的预算会增大 Agent 的实验能力**——它是在告诉 Agent"别为省成本而简化或放弃实验"，而不仅是一个上限。
- 除总量外，也可以把资源**分配到具体阶段**（例如"主实验最多用 4 张卡，verify 变体最多 2 张"），系统会按阶段分发。
- GPU 预算属于**硬约束**：Agent 会把每次实验规模压在预算内再启动，预算真的不够时会停下并上报，而不是偷偷超支。

### 1.3 声明硬约束与注意事项

`task.md` 中的内容，编排器会分成**两类**注入给各个子 Agent，并**只分发给与之相关的阶段**（针对性分发），而非一股脑塞给所有子 Agent。

**① 硬约束（`## HARD CONSTRAINTS`）**——三种：

| 类型 | 触发词 | 例子 |
|------|--------|------|
| **负向禁令** | 不准 / 不要 / 禁止 / 绝不 | "不要用 Pythia 2.8B""禁止对数据做子采样" |
| **强正向要求** | 必须 / 只能用 / 严格使用 / 就用 | "必须严格使用 Llama-3-8B""验证时只能用这两个模型" |
| **算力 / 资源分配** | 见 1.2 | GPU 小时 / 卡数预算、给某阶段分配的资源 |

子 Agent 应把硬约束当作不可逾越的红线——绝不超预算、绝不做被禁止的事、始终满足强正向指定；真的无法完成时会停下上报，而不是擅自突破。

**② 注意事项 / 告知（`## NOTICE`）**——你指明了但**没有强制语气**的内容，例如非强制的 model / dataset 选择、环境说明、偏好。系统会注入让相关子 Agent"知晓"，但以 Mechanist 在 Claim 阶段产出的实验计划 `EXPERIMENT_PLAN.md` 为权威；必要时 Agent 可以调整，但不会**擅自丢弃**——有冲突会上报而非自作主张。

**针对性分发（关键）**：约束可以**限定到某个阶段或某条 claim**，编排器只把它分发给对应的子 Agent。用自然语言点明范围即可，例如：

> 对 claim 3 进行 Verify 时只用 Pythia 1B 和 410M，暂时不要跑 2.8B。

这条只会进入 **verify 阶段**（以及负责写计划的 claim 阶段），**不会**误传给主实验。

### 1.4 声明进展通知

用自然语言表达通知意图即可，例如：

> 当实验取得进展时，向我的邮箱 example@gmail.com 发送通知，每小时同步一次。

启用后，流水线会在关键触点（实验完成 / verify 完成 / 全部结束 / halt / 需要人工介入）自动推送简报。不写通知意图时，通知功能完全静默。

注意：用户必须提前准备好消息通知渠道，Mechanist 只负责起草通知文本并扫描本地已配好的通知渠道，不会帮你安装通知渠道。

## 二、实验目录隔离

### 2.1 背景与问题

插件目前处于测试阶段。同一台机器上可能并存多个历史测试目录，其中包含之前测试的运行产物、他人复现相同实验的运行产物等。如果不加约束，Mechanist 在运行期间会聪明地读取这些历史目录中的内容，出现**跨实验借鉴**，导致测试结果失真。

因此需要确保每次测试运行的文件系统访问范围被限制在当前工作目录内，阻断 Agent 对历史测试目录的读取路径。

### 2.2 隔离方案

提供下面的两种机制控制 Mechanist 的文件访问范围，请任选一种：

**方案一：Prompt 软约束**

在 `task.md` 中规定：“禁止读取其它实验目录，禁止借鉴既往实验的数据、方案、组别设计等信息”。这样子，编排器在派发每个子 Agent 时，会显式要求 Agent 进行遵守。这是**提示词层面的约束**，依赖模型遵循指令。

**第二层：配置文件硬约束**

在**当前实验目录**下放一份 `.claude/settings.local.json`，用 `permissions.deny` 屏蔽掉所有其它历史实验目录的读取权限。

放置位置示例（假设本轮实验目录是 `exp/`）：

```
<project-dir>/
└── exp/
    └── .claude/
        └── settings.local.json     ← 只在这一轮实验目录里生效
```
示例：
```json
{
  "permissions": {
    "deny": [
      "Read(/absolute/path/to/verbal_confidence/**)",
      "Read(/absolute/path/to/verbal_confidence2/**)",
      "Read(/absolute/path/to/other_old_exp/**)"
    ]
  }
}
```

> [!WARNING]
> 这只是"软偏好"，不是硬边界。Claude 仍然**可以**通过 `Bash(cat ...)`、`Bash(head ...)`、`Grep`、`Glob` 等方式间接读到内容。但一般来说禁掉Read够用了。


使用要点：

- 路径必须是**绝对路径** + `/**` 通配后缀，才能匹配到目录下的全部文件。
- 每新增一轮实验，就在新目录里放一份新的 `settings.local.json`，把所有历史实验目录都追加到 `deny` 里。
- 该文件只对**在这个目录下启动的 Claude Code 会话**生效，不会影响其它项目。

## 三、实验行为观察与报告

以下是流水线运行时的有待进一步 check 的关键行为。请在你自己的实验中留意这些方面，发现问题及时反馈。

### 3.1 task.md 内容处理验证

编写好 `task.md` 并运行 `/auto` 后，验证其内容是否被正确解析和执行：

| 序号 | 观察项 | 说明 |
|------|--------|------|
| 3.1.1 | constraint → prompt 注入链路 | task.md 中声明的硬约束和注意事项是否出现在了对应子 Agent 的派发 Prompt 中？检查子 Agent 的执行记录，确认 `## HARD CONSTRAINTS` 和 `## NOTICE` 块的内容与 task.md 原文一致，且**只注入了相关阶段**（如 experiment 阶段的约束不应出现在 verify agent 的 Prompt 中）。 |
| 3.1.2 | GPU 预算是否被遵守 | task.md 中声明的 GPU 小时 / 卡数上限是否被 Agent 严格遵守？在 GPU 预算充足时，Agent 是否会以预算不足为理由拒绝进行实验？ |
| 3.1.3 | 硬约束是否被当真 | task.md 中的"不准 / 禁止 / 必须"等强语气约束，Agent 是否真的遵守了？ |

### 3.2 实验目录隔离

| 序号 | 观察项 | 说明 |
|------|--------|------|
| 3.2.1 | 实验目录隔离是否实际生效 | 如果启配置了实验目录隔离，观察 Agent 会不会偷偷借鉴既往的实验记录。如果是，则报告并重新实验。 |

### 3.3 编排器否定 Subagent 产出并触发迭代时的文档修改

#### 背景：设计之外的 emergent 行为

按照设计，流水线的四个阶段各由独立的 Subagent 负责——编排器只做派发与门控，不亲自对执行阶段进行检查。检查统一在 verify 阶段进行，在 iterate 阶段迭代重做。

但在测试中观察到了**设计之外**的行为：编排器会直接**否定（拒绝）Subagent 的工作成果**，并试图让同一个 Subagent 按新要求重做。这引发了两类问题：

**约束冲突**——编排器初次派发 Subagent 时注入了初始任务要求（约束 A）。当编排器拒绝成果后，向**同一个** Subagent 追加注入更新的任务要求（约束 B）。约束 B 通常与约束 A 及 Subagent 已产出的执行结果冲突。由于约束 A 是首次接收到的任务说明，地位相当于"System Prompt"，在 Subagent 内部通常强于后续追加的约束 B，导致 Subagent 陷入"该听初始任务 A 还是听追加任务 B"的矛盾。

**角色越界**——编排器检测到约束冲突后尝试与 Subagent 沟通。如果沟通无效，已观察到编排器会**亲自接管实验任务**（越过 Subagent 直接编辑实验脚本、提交 `/run-experiment`），打破了"编排器只管派发，Subagent 负责执行"的职责边界。

#### 约束方案

针对上述 emergent 行为，现已对编排器的重做流程进行了约束——编排器拒绝 Subagent 成果后，**不得向同一个 Subagent 追加注入修正指令**，而必须以磁盘上的 `EXPERIMENT_PLAN.md` 为唯一权威约束，走如下流程：

1. **终止**旧的 Subagent
2. 通过 **Claim Subagent** 更新 `EXPERIMENT_PLAN.md`（将修正需求写入计划的对应步骤）
3. 创建**新的** Experiment Subagent，仅传递更新后的 `EXPERIMENT_PLAN.md` 作为任务输入
4. 新 Subagent **只采信** `EXPERIMENT_PLAN.md` 的内容，不采信被拒那次运行遗留的旧脚本和旧文档

#### 观察要点

| 序号 | 观察项 | 说明 |
|------|--------|------|
| 3.3.1 | 编排器拒绝 Subagent 成果的频率 | 在当前测试中，编排器直接否定 Subagent 工作成果的情况出现得多不多？什么场景下容易触发？ |
| 3.3.2 | 拒绝后是否按约束流程执行 | 编排器拒绝 Subagent 成果后，是否遵循了上述约束流程（终止旧 Subagent → 更新计划 → 创建新 Subagent），还是仍然走了"向旧 Subagent 追加注入"或"亲自接管"的旧路径？ |
| 3.3.3 | 新 Subagent 是否只采信计划 | 新创建的 Subagent 是否仅依据 `EXPERIMENT_PLAN.md` 执行，没有读取被拒那次运行遗留的旧文档？ |
| 3.3.4 | 覆盖语义是否正确 | 修正后，`EXPERIMENT_RESULTS.md` 中是否用 `superseded: <旧 run-id>` 标记了被替代的结果，而非同时保留新旧两份互相矛盾的叙事？ |
| 3.3.5 | 磁盘文件是否正确 | 新创建的 Subagent 是否会将被拒那次运行遗留的旧文档和旧数据进行覆写或归档，避免影响新文档和新数据？ |
